# Canon CLI Automation Contract

## Problem

Agents and CI jobs need a stable local Canon CLI surface that can be parsed without relying on human text. The existing CLI exposes JSON for semantic checks, but `status`, `ingest`, and `render` still require string parsing.

## Desired Behavior

- `status --json` emits stable snake_case JSON for repository health, spec counts, domain counts, ledger counts, and layout repair metadata.
- `ingest --json` emits the created `spec_id`, `spec_path`, `ledger_path`, and resolved `parents` from the ingest result.
- `render --json` emits render write counts plus AI mode, provider, applied/fallback status, and fallback reason.
- Existing human-readable default output remains unchanged for `status`, `ingest`, and `render`.
- JSON output is produced with structured Go values and `encoding/json`.
- CLI users can run a minimal CI loop with `init --ai off`, `ingest --json`, `check --json`, `render --write --json`, `status --json`, and `semantic-diff --json`.

## Success Criteria

- Unit and command tests parse JSON output for `status --json`, `ingest --json`, and `render --json` and assert key fields.
- Existing human-output tests continue to pass.
- Documentation states that success exits with code 0 and validation, argument, gate, and layout failures exit with code 1.
- Manual CLI evidence includes one positive automation flow and one related negative flow with parseable JSON where the failed command supports JSON.
