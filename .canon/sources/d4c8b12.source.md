# Init Review-First Acceptance

## Problem

`canon init` can generate bootstrap specs from AI project context. Non-interactive execution previously treated generated specs as accepted implicitly, which allowed `--response-file` replay or headless runs to canonicalize generated specs into `.canon/` without a human review action or explicit bulk-accept flag.

## Desired Behavior

Generated init specs must become canonical only through one of two acceptance paths:

1. Interactive review where the user accepts an individual spec.
2. Explicit bulk acceptance through `--accept-all` or compatible `--no-interactive`.

Non-interactive init runs that generate specs without explicit bulk acceptance must fail before writing `.canon/specs/` or `.canon/ledger/` entries, and the error must tell the user to rerun interactively or pass an explicit acceptance flag.

## Scope

In scope:

1. `cmd/canon init` flag handling.
2. `internal/canon.InitOptions` acceptance modeling.
3. Init unit and CLI tests for explicit acceptance, non-interactive rejection, skip, quit, malformed response, and invalid AI mode behavior.
4. README documentation for review-first bootstrap behavior and `specs/init-drafts/` preservation.

Out of scope:

1. A separate bootstrap command.
2. Changing AI response schema.
3. Moving skipped or deferred drafts into `.canon/`.

## Success Criteria

1. `canon init --response-file <file> --accept-all` ingests generated specs and writes ledger entries.
2. `canon init --response-file <file>` in a non-TTY shell fails without writing canonical specs or ledger entries.
3. Interactive skip and quit keep generated drafts under `specs/init-drafts/` and outside `.canon/specs/`.
4. Malformed response files and unsupported init AI modes remain rejected with clear errors.
