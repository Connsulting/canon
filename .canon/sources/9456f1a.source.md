# Canon Log CLI Validation and Scope Fixes

## Problem Statement
`canon log` has two localized CLI bugs:
- it silently accepts unexpected positional arguments instead of rejecting them
- `canon log --date absolute --all=false` can ignore the explicit primary-head scope request because the legacy detailed-output path bypasses structured scoping

## Scope
- In scope: targeted `canon log` fixes in `cmd/canon/main.go`, focused regression tests, and any minimal documentation clarification needed for `log` scoping defaults
- Out of scope: changes to unrelated commands, broad log rendering refactors, or any change to the default `canon log` output contract beyond documented clarification

## Desired Behavior
- `canon log` rejects positional arguments with a clear CLI error
- plain `canon log` keeps its current default behavior:
  - detailed output
  - relative dates
  - all disconnected heads included by default
- `canon log --date absolute --all=false` respects the explicit primary-head scope request even when the detailed absolute-date path is selected
- detailed absolute-date output keeps the existing `Type:` and `Domain:` lines whether it is rendered through the legacy or structured scope-aware path

## Compatibility Expectations
- Preserve the existing legacy detailed absolute-date output for the default/all-head case
- Preserve the detailed absolute-date field set when `--all=false` forces scope-aware rendering
- Only route around the legacy path when needed to honor an explicit narrowed scope
- Do not change graph, oneline, grep, domain, type, color, or tag behavior outside this fix

## Acceptance Criteria
- `canon log unexpected` fails with a positional-argument error
- `canon log --date absolute --all=false` only prints the primary head and its reachable dependencies
- `canon log --date absolute --all=false` keeps the detailed `Type:` and `Domain:` fields
- `canon log --date absolute` keeps the legacy detailed fields, including `Type:` and `Domain:`
- `canon log` still prints relative dates and includes all disconnected heads by default
- Focused unit tests cover each behavior above

## Verification
1. Format touched Go files with `gofmt -w`.
2. Run `go test ./cmd/canon ./internal/canon`.
3. Positive CLI check:
   `go run ./cmd/canon log --root <repo-or-fixture> --date absolute --all=false`
4. Negative CLI check:
   `go run ./cmd/canon log unexpected`

## Risks / Tradeoffs
- The command currently has a legacy rendering fast path; the safest fix is to narrow when that path is used instead of rewriting rendering logic.
- Documentation must be explicit that `--all` defaults to including all disconnected heads, with `--all=false` as the opt-in narrowed scope.
