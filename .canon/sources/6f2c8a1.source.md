# Privacy Policy Consistency Checker Draft Spec

## Problem Statement
Canon currently has no command that compares repository implementation against stated privacy policy claims. Teams need a CI-friendly check that can surface likely contradictions between policy language and code.

## Desired Behavior
Add an AI-first `canon privacy-check` command that loads a local policy file, evaluates selected repository code context, and returns structured findings with optional severity threshold enforcement.

## Scope (v1)
In scope:
1. Static repository analysis only (local files; no runtime probing).
2. Required local policy file input.
3. Scoped code context collection via repeatable `--code-path`.
4. AI adjudication modes: `auto` and deterministic `from-response`.
5. Structured findings with status + severity and deterministic normalization/sorting.
6. JSON and text outputs.
7. CI threshold gating via `--fail-on`.

Out of scope:
1. Dynamic runtime telemetry validation.
2. Legal interpretation guarantees.
3. External services, network traces, or policy website fetching.

## CLI Contract
Command:
- `canon privacy-check [--root <path>] --policy-file <path> [--code-path <path> ...] [--context-limit <bytes>] [--max-file-bytes <bytes>] [--ai <mode>] [--ai-provider <name>] [--response-file <path>] [--json] [--fail-on <severity>]`

Flags:
1. `--root <path>` repository root (default `.`).
2. `--policy-file <path>` required local policy file path under root.
3. `--code-path <path>` repeatable include scope for code context.
4. `--context-limit <bytes>` cap total context bytes.
5. `--max-file-bytes <bytes>` cap per-file bytes prior to truncation.
6. `--ai <mode>` one of `auto`, `from-response`.
7. `--ai-provider <name>` `codex` or `claude` (default from config).
8. `--response-file <path>` precomputed response for deterministic runs.
9. `--json` machine-readable output.
10. `--fail-on <severity>` threshold (`low`, `medium`, `high`, `critical`).

Validation:
1. Reject positional args.
2. Require `--policy-file`.
3. Reject invalid mode or fail-on values.
4. Require `--response-file` in `from-response` mode.

Exit behavior:
1. Non-zero on validation/analyzer errors.
2. Non-zero when threshold is met/exceeded.

## Result Model (v1)
Finding fields:
1. `claim_key`
2. `claim`
3. `status` (`supported`, `contradicted`, `unknown`)
4. `severity` (`none`, `low`, `medium`, `high`, `critical`)
5. `message`
6. `evidence[]`

Summary fields:
1. total findings
2. highest severity
3. counts by severity
4. counts by status

Result metadata:
1. root path
2. policy file path
3. code path count
4. context file count
5. context byte totals/limits
6. optional fail-on and threshold-exceeded state

## Determinism and Normalization
1. Normalize/trim AI fields.
2. Default missing values where possible.
3. Drop invalid entries.
4. Deduplicate exact duplicate findings.
5. Stable sort by severity, status, and claim key.
6. Deterministic summary counts.

## Acceptance Criteria
1. New CLI command wired with full flag contract.
2. Core analyzer implemented in `internal/canon/privacy_policy.go`.
3. Types and CLI API wrappers added.
4. Unit tests cover validation, parsing, normalization/dedupe/sort, summary, and threshold logic.
5. CLI tests cover success, JSON shape, missing required flags, invalid fail-on, positional args, and threshold failure.
6. README documents command/options/examples.
7. Manual positive and negative CLI checks run.

## Validation Plan
1. `gofmt -w` on changed Go files.
2. `go test ./...`.
3. `go vet ./...`.
4. Manual positive:
   - `go run ./cmd/canon privacy-check --root . --policy-file <path> --response-file <path>`
5. Manual negative:
   - `go run ./cmd/canon privacy-check --root . --policy-file <path> --response-file <path> --fail-on medium` (when response highest severity is high)
