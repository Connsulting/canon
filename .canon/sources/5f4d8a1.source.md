# Logging Quality Auditor (Draft)

## Problem Statement
Canon stores history and provenance in `.canon/ledger`, `.canon/specs`, and `.canon/sources`. Today there is no dedicated analyzer that audits those artifacts for completeness, referential integrity, and metadata quality. Teams need a deterministic way to detect broken or degraded logging artifacts before they affect CI or downstream workflows.

## Scope (v1)
`logging-audit` audits Canon artifacts under `.canon/` only.

In scope:
- Ledger record field completeness and parseability.
- Referential integrity across ledger/spec/source artifacts.
- Timestamp and sequence validity checks.
- Metadata quality checks (type/title/domain coherence and hash consistency).
- Deterministic text and JSON output.
- Optional CI fail threshold via `--fail-on`.

Out of scope:
- Runtime application logger instrumentation.
- Non-Canon logging systems.
- AI adjudication in this command (deterministic offline analyzer only).

## Assumptions
- Repository root is provided with `--root` (default `.`).
- Canon artifacts live under `.canon/ledger`, `.canon/specs`, `.canon/sources`.
- Ledger entries are JSON files with `LedgerEntry` schema.
- Severity ranking is `none < low < medium < high < critical`.

## CLI Contract
Command:
- `canon logging-audit [--root <path>] [--json] [--fail-on <severity>]`

Flags:
- `--root <path>`: repository root to audit (default `.`).
- `--json`: emit machine-readable JSON payload.
- `--fail-on <severity>`: fail command when highest finding severity meets/exceeds threshold (`low`, `medium`, `high`, `critical`).

Validation:
- Positional arguments are rejected.
- Invalid `--fail-on` values are rejected with actionable error text.

Exit behavior:
- `0` when audit completes and threshold is not exceeded.
- Non-zero when input validation fails, audit execution fails, or threshold is exceeded.

## Checks

### 1) Completeness
- Required ledger fields are present and non-empty: `spec_id`, `title`, `type`, `domain`, `sequence`, `ingested_at`, `content_hash`, `spec_path`, `source_path`.
- `sequence` must be positive.
- `ingested_at` must parse as RFC3339.

### 2) Referential Integrity
- `spec_path` and `source_path` must be relative canonical paths under `.canon/specs` and `.canon/sources`.
- Referenced spec/source files must exist.
- Parent ids in `parents` must resolve to existing ledger `spec_id` values.
- Ledger `spec_id` should map to matching spec metadata where available.

### 3) Sequence/Time Quality
- Duplicate `sequence` values are flagged.
- Sequence should be monotonic with ingest time ordering.

### 4) Metadata Quality
- Ledger `type`, `title`, `domain` should align with parsed spec frontmatter.
- Ledger `content_hash` should match current spec file content hash.
- Unknown spec type values and weak metadata are flagged.

## Severity Model
- `critical`: cannot parse ledger artifact / unreadable artifact.
- `high`: broken integrity likely to invalidate history traversal (missing references, invalid required fields, invalid paths, hash mismatch).
- `medium`: likely quality issue requiring remediation (metadata mismatch, unknown type, non-monotonic sequence).
- `low`: hygiene/quality gap with lower impact.

## Output Contract
Result includes:
- Root and audited directories.
- Artifact counts (ledger/spec/source files).
- Ordered findings with deterministic sort key.
- Summary (total findings, highest severity, severity counts).
- Threshold metadata (`fail_on`, `threshold_exceeded`) when requested.

Determinism requirements:
- Findings sorted by severity rank desc, then stable lexical keys.
- Text output ordering mirrors JSON ordering.

## Acceptance Criteria
- New `canon logging-audit` command exists and appears in help/docs.
- JSON output is valid and stable for a fixed repo state.
- Threshold behavior is enforced consistently in text and JSON modes.
- Unit tests cover core normalization, ordering, summary, and threshold logic.
- CLI tests cover JSON shape, threshold failure, invalid `--fail-on`, positional argument rejection.
