# PII Exposure Scanner Draft Spec

## Problem Statement
Canon currently has no repeatable repository audit that flags likely PII exposure across source, config, data, and operational surfaces. Teams need a deterministic scan-and-triage workflow to identify and remediate high-confidence leaks before merge.

## Desired Behavior
Run a repository-local PII exposure audit that:
1. Finds hardcoded PII-like literals in code/config/data.
2. Finds log/error statements that may emit PII without redaction.
3. Finds plaintext secrets in env/config/CI surfaces.
4. Finds plaintext storage of sensitive user data.
5. Finds `.gitignore` gaps and tracked sensitive files.

## Scope
In scope:
1. Text file scanning of tracked repository content and standard config paths.
2. Pattern + context based triage for signal over noise.
3. Structured findings output with file, line, category, severity, detail, recommendation.
4. Optional patching of confirmed high-confidence risks.

Out of scope:
1. Runtime traffic inspection.
2. External secret-manager introspection.
3. Decrypting binary artifacts.

## Inputs and File Coverage
1. Primary corpus: `git ls-files`.
2. Extra surfaces: `.env*`, `.github/workflows/*`, `docker-compose*`, `config*.{yaml,yml,json,toml,env}` if present.
3. Structured data emphasis: `*.json`, `*.yaml`, `*.yml`, `*.csv`, `*.sql`, fixtures/seeds/testdata.
4. Exclude vendored/third-party code (e.g., `vendor/`, external generated bundles).

## Detection Categories
1. `hardcoded-pii`
   - Email, phone, SSN, credit-card-like numbers, IP identifiers, address-like literals, DOB-like dates, passport/driver-license-like IDs, structured full names.
2. `pii-in-logs`
   - Logging and error formatting APIs where fields such as `email`, `name`, `phone`, `address`, `ssn`, `token`, `password` appear in emitted text.
3. `env-secret`
   - Plaintext API keys/tokens/passwords in env/config/CI definitions.
4. `unencrypted-storage`
   - Plaintext persistence of sensitive fields without hashing/encryption/masking.
5. `gitignore-gap`
   - Missing ignore coverage for `.env*`, `*.pem`, `*.key`, `credentials.*`, `secrets.*`, and sensitive dump extensions.

## Severity Rubric
1. `critical`: plaintext high-impact credentials or direct production-grade PII exposure.
2. `high`: clear sensitive-data leakage vector (logs/storage/config) likely exploitable.
3. `medium`: realistic-looking PII in fixtures/test artifacts or moderate exposure risk.
4. `low`: weak signal requiring confirmation or policy-hardening only.

Rule: realistic-looking test fixture PII is `medium` by default.

## Output Schema
For each finding:
1. `file`: repo-relative path.
2. `line`: line number(s).
3. `category`: one of `[hardcoded-pii, pii-in-logs, env-secret, unencrypted-storage, gitignore-gap]`.
4. `severity`: one of `[critical, high, medium, low]`.
5. `detail`: what was found and why it is risky.
6. `recommendation`: concrete remediation.

Summary:
1. total findings by category.
2. total findings by severity.

## Verification and Acceptance
1. Record reproducible scan commands and triage decisions.
2. Apply remediations for confirmed high-confidence findings.
3. Run `go test ./...`.
4. Run at least one positive and one negative end-to-end CLI command related to the implemented behavior.
5. Ingest finalized spec/results into `.canon/` artifacts; do not commit this draft spec.

## Risks and Tradeoffs
1. Regex-only scanning may over-report without context triage.
2. Highly generic number patterns can create false positives.
3. Restricting to tracked text files may miss ignored local leaks; mitigated by `.gitignore` audit.

## Audit Results (2026-03-07)
Findings:
1. `.gitignore` lacked baseline protections for `.env*`, key material, credential files, and common data-dump extensions.

Remediation applied:
1. Added ignore coverage for `.env`, `.env.*`, `*.pem`, `*.key`, `credentials.*`, `secrets.*`, `*.csv`, `*.xlsx`, and `*.sql`.
2. Added SQL migration allow-list exceptions for `db/migrations/*.sql` and `**/migrations/*.sql`.

Post-remediation verification:
1. No tracked sensitive candidate files matched (`.env*`, keys, credentials, `*.sql`, `*.csv`, `*.xlsx`).
2. Pattern and contextual scans found no high-confidence hardcoded PII, log leakage, or plaintext secret storage in tracked files.
