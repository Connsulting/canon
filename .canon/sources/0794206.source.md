# Test Flakiness Analyzer (Draft)

## Problem Statement
Go test suites can hide unstable tests that pass in one run and fail in another. This creates noisy CI failures and reduces confidence in test outcomes. We need a deterministic CLI analyzer that surfaces pass/fail instability quickly.

## Scope (v1)
- Analyze Go tests only.
- Execute repeated `go test -json -count=1` runs.
- Aggregate terminal test outcomes (`pass`, `fail`, `skip`) per test across runs.
- Classify a test as flaky only when it has both pass and fail outcomes.
- Produce deterministic text and JSON output.
- Support opt-in CI gating via `--fail-on-flaky`.

Out of scope (v1):
- Non-Go project support.
- Historical result storage across command invocations.
- Probabilistic/heuristic flakiness scoring.
- Treating skip-only variance as flaky.

## CLI Contract
Command:
- `canon test-flakiness`

Options:
- `--root <path>`: repository root (default `.`)
- `--runs <n>`: repeated test runs (minimum `2`, default `4`)
- `--package <pattern>`: Go package pattern(s), repeatable (default `./...`)
- `--json`: emit machine-readable JSON
- `--fail-on-flaky`: fail command when flaky tests are found

Behavior:
- No positional arguments allowed.
- `--runs < 2` is a validation error.
- Deterministic ordering in output (sorted by package/test).

## Data Model (v1)
- Options include run count, package scope, and fail gate flag.
- Result includes:
  - absolute root
  - run count
  - package scope
  - flaky findings with per-test outcome counts
  - summary counts (total, flaky, stable pass, stable fail, skip-only)
  - optional fail gate metadata (`enabled`, `exceeded`)

## Risks and Tradeoffs
- Re-running tests increases runtime; default run count is intentionally small.
- Running real tests may execute side effects in user test suites.
- Package-level build failures with no test events should return actionable errors.
- `go test -json` output may include non-JSON lines; parser must ignore noise safely.

## Acceptance Criteria
1. `canon test-flakiness` runs repeated Go tests and reports flaky tests (pass+fail across runs).
2. Text and JSON outputs are deterministic.
3. `--fail-on-flaky` returns non-zero when flaky tests exist.
4. `--runs 1` returns validation error.
5. Unit tests cover parser, aggregation/classification determinism, validation, and summary math.
6. CLI tests cover JSON output shape, flaky detection, fail gate behavior, and negative validation.

## Validation Checklist
- [ ] `gofmt -w` on changed Go files
- [ ] `go test ./...`
- [ ] `go vet ./...`
- [ ] Manual positive E2E: flaky fixture with JSON output shows flaky finding
- [ ] Manual negative E2E: `--runs 1` fails validation
