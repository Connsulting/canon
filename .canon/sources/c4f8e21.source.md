# PII Exposure Scanner

## Scope

Implement a deterministic offline repository scanner exposed via `canon pii-scan`.

### Included

- Scan repository text files (source, config, scripts, data files) for:
  - hardcoded PII literals
  - PII in logs/errors
  - env/secret exposure
  - unencrypted plaintext storage patterns
  - `.gitignore` hygiene gaps and tracked sensitive files
- Exclude vendored/third-party directories (`vendor`, `third_party`, `third-party`, `node_modules`).
- Emit deterministic findings sorted by severity/file/line.
- Emit summary totals by category and severity.
- Support `--json` and `--fail-on` thresholds.

### Required Finding Schema

Each finding must include:

- `file`
- `line`
- `category`
- `severity`
- `detail`
- `recommendation`

### Severity Policy

- `critical`: raw secrets, SSN/card-like literals, plaintext high-risk storage
- `high`: likely sensitive PII exposure
- `medium`: moderate risk and realistic-looking PII in fixtures/test data
- `low`: hygiene gaps with lower immediate exposure risk

## Key Detection Rules

### hardcoded-pii

- Email, phone, SSN, credit-card-like numbers, IP address identifiers, street addresses, DOB patterns, passport/driver-license-like IDs.
- Structured full-name detection in JSON/YAML/CSV/SQL data contexts.

### pii-in-logs

- Log/error interpolation in `fmt.Sprintf`, `fmt.Errorf`, `log.Printf`, `slog`, `zerolog`, `zap`, `logrus`, `errors.Wrap` and similar.
- Multiline call support for interpolated arguments.
- Suppress when redaction/masking indicators are present.

### env-secret

- `.env*`, config files, docker-compose files, CI workflow env blocks.
- Detect plaintext secret assignments including docker-compose environment list syntax (`- KEY=value`).

### unencrypted-storage

- SQL schema/value patterns, struct fields, and write paths persisting PII in plaintext.

### gitignore-gap

- Ensure recommended ignore coverage for `.env*`, `*.pem`, `*.key`, `credentials.*`, `secrets.*`, `*.sql`, `*.csv`, `*.xlsx`.
- Flag tracked sensitive files.

## Non-goals

- No network calls.
- No AST or deep semantic parsing.
- No auto-remediation.
- No cryptographic verification of encryption-at-rest implementation.

## Validation

- Unit tests for each category and edge-case regressions.
- CLI tests for JSON contract and threshold failures.
- Manual end-to-end commands:
  - Positive: `go run ./cmd/canon pii-scan --root . --json`
  - Negative: `go run ./cmd/canon pii-scan --fail-on urgent`
