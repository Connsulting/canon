# Metrics Coverage Analyzer

## Problem Statement
Canon currently lacks a deterministic analyzer that answers whether Go CLI command handlers are instrumented with metrics calls. Teams need a static coverage signal they can run locally and in CI without runtime telemetry dependencies.

## Scope
- Add a new CLI analyzer command: `metrics-coverage`.
- Analyze Go CLI command handlers statically (AST based, offline).
- Produce deterministic text and JSON output with summary statistics.
- Support CI threshold gating with `--fail-under`.

## Assumptions
- Coverage is inferred from explicit metric-emitter call patterns inside handler AST bodies.
- Runtime telemetry delivery is out of scope.
- v1 targets Go handler functions named `cmd*` with signature `func([]string) error` in `cmd/` sources.

## CLI Contract
```text
canon metrics-coverage [options]

Options:
  --root <path>        repository root (default: ".")
  --json               output machine-readable JSON
  --fail-under <pct>   fail when coverage percent is below threshold [0..100]
```

Rules:
- Reject positional arguments.
- Reject invalid `--fail-under` values (non-numeric, <0, >100).
- Return non-zero when `--fail-under` is provided and coverage is below the threshold.

## Heuristic Rules (v1)
- Target handlers:
  - Top-level functions in Go files under `cmd/`
  - Name prefix `cmd` with uppercase suffix (for example `cmdLog`)
  - Signature exactly `func([]string) error`
- Instrumentation detection:
  - Traverse handler AST and inspect call expressions
  - Treat a call as instrumentation when callee path includes metrics context token(s) (for example `metric`, `metrics`, `telemetry`, `stats`, `meter`, `prometheus`) and an emitter verb (for example `record`, `emit`, `inc`, `add`, `observe`, `count`, `set`, `timing`)
- Determinism:
  - Stable file traversal and sorted findings
  - Stable missing-target list
  - Coverage rounded to 2 decimals

## Output Schema
```json
{
  "root": "/abs/path",
  "findings": [
    {
      "handler": "cmdLog",
      "file": "cmd/canon/main.go",
      "line": 123,
      "instrumented": false,
      "matched_calls": []
    }
  ],
  "missing_targets": ["cmdLog (cmd/canon/main.go:123)"],
  "summary": {
    "target_count": 10,
    "instrumented_count": 6,
    "missing_count": 4,
    "coverage_percent": 60.0
  },
  "fail_under": 75.0,
  "threshold_exceeded": true
}
```

## Acceptance Criteria
- New command `metrics-coverage` is available and listed in help/docs.
- Analyzer returns deterministic results for the same tree.
- JSON output includes findings, summary, missing targets, and threshold state.
- Threshold gating is enforced via `--fail-under`.
- Unit tests and CLI tests cover detection, ordering, invalid inputs, empty-target behavior, and threshold semantics.

## Test Plan
Unit tests:
- Heuristic detection for instrumented vs missing handlers.
- Deterministic ordering of findings/missing targets.
- Empty/no-target behavior.
- Invalid root path handling.
- Threshold logic (`below`, `equal`, `above`).

CLI tests:
- JSON output shape for `metrics-coverage --json`.
- Threshold failure returns error and still prints summary.
- Invalid `--fail-under` is rejected.
- Positional args are rejected.

Manual end-to-end checks:
- Positive: `go run ./cmd/canon metrics-coverage --root .`
- Positive: `go run ./cmd/canon metrics-coverage --root . --json`
- Negative: `go run ./cmd/canon metrics-coverage --root . --fail-under 101`
- Negative: positional argument rejection

## Risks and Tradeoffs
- Static heuristics can produce false positives/negatives for custom wrappers.
- Restricting targets to conventional handler signatures keeps behavior predictable for v1.
- Future versions can expand target detection and metric-call pattern support.
