# Frontmatter Round-Trip Parser Draft Spec

## Problem Statement
Canon emits YAML-like frontmatter for specs, but the handwritten parser does not round-trip all Canon-generated values safely. Quoted scalars lose escaping and bracket lists split on every comma, which can corrupt metadata when specs are reloaded from canonical files.

## Scope
- In scope: frontmatter parsing behavior for Canon-generated spec files in `internal/canon/parse.go`.
- In scope: targeted unit and CLI regression coverage for quoted titles and quoted list items.
- Out of scope: expanding support to full YAML semantics or changing unrelated spec ingestion behavior.

## Desired Behavior
Canon-generated frontmatter must round-trip through `canonicalSpecText` and `parseSpecText` without corrupting quoted metadata.

This includes:
1. Titles containing quotes, backslashes, colons, and commas.
2. Bracket list fields containing quoted values with commas or escaped characters.
3. Continued rejection of malformed quoted scalars and malformed lists.

## Acceptance Criteria
1. `parseSpecText(canonicalSpecText(spec))` preserves quoted scalar content and quoted list values for Canon-emitted frontmatter.
2. Malformed quoted frontmatter still returns an error.
3. Existing parser defaults remain intact for omitted optional metadata.
4. Required-field validation still rejects invalid frontmatter.
5. CLI regression coverage proves human-readable metadata is preserved after ingest and show/log flows.

## Validation Plan
1. `gofmt -w` on changed Go files.
2. `go vet ./...`
3. `go test ./...`
4. Positive CLI check:
   `canon init --ai off`, `canon ingest` on a quoted-frontmatter spec, then `canon show`/`canon log`.
5. Negative CLI check:
   ingest malformed frontmatter and confirm the command fails.
