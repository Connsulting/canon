# Dependency Risk Scanner Draft Spec

## Problem Statement
Canon currently has no built-in way to assess dependency risk for Go modules. Teams need a deterministic and offline command they can run in CI and locally to identify security and maintenance hygiene issues without relying on external CVE services.

## Desired Behavior
Add a `deps-risk` CLI command that analyzes Go dependencies from `go.mod` and optional `go.sum`, emits structured findings in security and maintenance categories, supports JSON output, and allows CI gating via a fail-on severity threshold.

## Scope (v1)
In scope:
1. Analyze only Go module manifests in target root (`go.mod` and optional `go.sum`).
2. Use deterministic offline heuristics (no network calls).
3. Report findings with severity and category metadata.
4. Provide stable human-readable and JSON outputs.
5. Support CLI fail thresholds (for CI) via `--fail-on`.

Out of scope:
1. External vulnerability database lookups.
2. Language ecosystems other than Go.
3. Auto-remediation actions (for example dependency updates).

## Assumptions
1. This repo contains a root `go.mod` and may or may not contain `go.sum`.
2. Heuristic risk signals still provide immediate value (supply-chain and maintenance hygiene).
3. Deterministic output ordering is required for reliable tests and CI.

## CLI Contract
Command: `canon deps-risk`

Flags:
1. `--root <path>` (default `.`): repository root to scan.
2. `--json`: emit machine-readable JSON result.
3. `--fail-on <severity>`: non-zero exit when max finding severity meets/exceeds threshold (`low`, `medium`, `high`, `critical`).

Validation:
1. Reject positional arguments.
2. Reject invalid severity strings for `--fail-on`.

Exit behavior:
1. Command execution errors return non-zero.
2. When `--fail-on` is set and threshold is exceeded, command returns non-zero.

## Risk Model (Heuristics)
Findings are emitted per risk signal:
1. Replace to local path (`replace => ../path` or absolute path): high severity security signal.
2. Replace to alternate module path (fork/remap): medium severity security signal.
3. Required module pinned to `v0.*`: medium severity maintenance signal.
4. Required module pinned to pseudo-version (`v0.0.0-yyyymmddhhmmss-<sha>`): medium severity maintenance signal.
5. Required module pinned to pre-release (`-alpha`, `-beta`, `-rc`, etc.): low severity maintenance signal.
6. Missing `go.sum` when dependencies exist: high severity security signal.

Severity order: `none < low < medium < high < critical`.

## Determinism Requirements
1. Sort findings by descending severity, then category, module, and rule id.
2. Stable summary fields (`total findings`, `counts by severity`, `highest severity`).
3. JSON output uses canonical struct fields and deterministic ordering of findings.

## Acceptance Criteria
1. `deps-risk` runs successfully on a repo with `go.mod`.
2. Human output includes summary and finding lines when findings exist.
3. `--json` output is valid JSON with findings and summary metadata.
4. `--fail-on` returns non-zero when threshold is reached/exceeded.
5. Invalid `--fail-on` values are rejected with clear error.
6. Unit tests cover heuristics, severity aggregation, and empty dependency cases.
7. CLI tests cover success, threshold failure, and invalid input handling.

## Validation Plan
1. `gofmt -w` on changed Go files.
2. `go test ./...`.
3. `go vet ./...`.
4. Manual positive CLI check in this repo:
   - `go run ./cmd/canon deps-risk --root .`
5. Manual negative CLI check in this repo:
   - `go run ./cmd/canon deps-risk --fail-on invalid`
