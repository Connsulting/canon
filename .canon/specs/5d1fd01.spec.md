# Semantic Diff Explainer (Draft)

## Problem Statement
Canon currently supports semantic checks (`check`, `blame`) and risk scanners (`deps-risk`, `schema-evolution`), but it lacks a focused command for explaining the semantic meaning of raw code diffs. Teams reviewing pull requests need a concise explanation of behavior and risk impact, not just line-level patches.

## Desired Behavior
Add a new AI-first CLI command:

- `canon semantic-diff`

The command should:

1. Accept unified diff input from either:
- `--diff-file <path>`
- default `git diff --no-ext-diff --unified=0 -- .` when `--diff-file` is absent

2. Produce:
- Human-readable terminal output by default.
- Machine-readable JSON with `--json`.

3. Use AI as the semantic adjudication source:
- `--ai auto` runs provider-backed structured generation.
- `--ai from-response` replays deterministic fixtures from `--response-file`.

4. Normalize output deterministically:
- Stable ordering of changed files, explanations, category counts, and evidence.
- Deduplicate semantically identical explanation/evidence entries.

## Minimal v1 Scope / Assumptions
- No language-specific AST/CFG analysis in v1.
- Semantic interpretation is LLM-driven via prompt + JSON schema constraints.
- Deterministic testability relies on `--response-file` replay.
- File-level and hunk-level evidence references are sufficient for v1.

## CLI Contract

### Command
- `canon semantic-diff [flags]`

### Flags
- `--root <path>`: repository root (default `.`)
- `--diff-file <path>`: unified diff input file
- `--json`: emit JSON output contract
- `--ai <mode>`: `auto|from-response` (default `auto`)
- `--ai-provider <name>`: `codex|claude` (default from config)
- `--response-file <path>`: AI response fixture (required for `from-response`)

### Exit/Errors
- Non-zero on invalid flags, invalid input, unresolved files, unsupported mode/provider, malformed AI response, or runtime execution failures.
- Non-zero on positional arguments (command accepts none).

## JSON Output Contract

Top-level:
- `root`
- `diff_source`
- `diff_bytes`
- `changed_file_count`
- `total_added_lines`
- `total_deleted_lines`
- `total_hunks`
- `changed_files[]`
- `explanations[]`
- `summary`

`changed_files[]`:
- `file`
- `status` (`added|deleted|modified|renamed`)
- `added_lines`
- `deleted_lines`
- `hunk_count`

`explanations[]`:
- `id`
- `category`
- `impact` (`low|medium|high|critical`)
- `summary`
- `rationale`
- `evidence[]`

`evidence[]`:
- `file`
- `kind` (`file|hunk`)
- optional: `old_start`, `old_lines`, `new_start`, `new_lines`

`summary`:
- `ai_model`
- `ai_summary`
- `total_explanations`
- `highest_impact`
- `impact_counts`
- `category_counts`

## Validation / Error Semantics
- Reject unsupported `--ai` values with explicit guidance.
- `from-response` mode must fail when `--response-file` is missing.
- Empty diff input fails with `no diff content found for semantic-diff`.
- Invalid response JSON fails with `invalid AI semantic-diff response JSON`.
- Unavailable provider in `auto` fails with `ai provider <name> is not runtime-ready`.

## Test Matrix

### Unit Tests (`internal/canon/semantic_diff_test.go`)
- Diff parsing from file with mixed status changes.
- Deterministic normalization of explanations/evidence/category counts.
- Default git-diff path when `--diff-file` is omitted.
- Validation failures:
- unsupported mode
- missing `--response-file` in `from-response`
- empty diff
- malformed AI response
- unsupported runtime provider

### CLI Tests (`cmd/canon/main_test.go`)
Positive:
- Text output with `--response-file` fixture.
- JSON output shape with `--json --response-file`.

Negative:
- Invalid `--ai` mode rejected.
- Positional args rejected.
- Missing response file in `--ai from-response` rejected.
- Invalid/missing diff file path returns failure.

## Success Criteria
- `semantic-diff` command is discoverable in `help`/README.
- Both text and JSON contracts are stable and test-covered.
- Unit tests and CLI tests pass.
- Manual positive and negative CLI runs pass.
