# Init Gitignore-Aware Scan Draft Spec

## Problem Statement
`canon init` builds AI scan context from repository files, but its current ignore handling is only a partial `.gitignore` approximation. That can leak files Git would ignore into the directory tree and prompt context, especially around ignored directories, wildcard patterns, nested ignore rules, and negation edge cases.

## Desired Behavior
When `canon init` scans a Git-backed repository, it should exclude any path Git would ignore according to the repository's ignore rules before building the scan tree and file context.

## Scope
In scope:
1. Exclude Git-ignored files from `canon init` scan context and directory tree output.
2. Preserve `--include` as an explicit override for ignored paths.
3. Preserve `--exclude` as an explicit exclusion layer.
4. Keep a non-Git fallback so `canon init` still works in directories without Git metadata.
5. Add unit coverage for ignored directories, wildcard patterns, and negation semantics that do not reopen ignored parent directories.

Out of scope:
1. Changing non-`init` commands.
2. Reworking AI prompt structure beyond removing ignored paths.
3. Full performance optimization for every possible giant ignored directory outside existing scan pruning rules.

## Assumptions
1. In real repositories, Git itself is the source of truth for ignore semantics.
2. Exact ignore behavior matters more than preserving the current approximate matcher in Git-backed repos.
3. Non-Git directories should retain best-effort `.gitignore` support instead of failing `init`.

## Acceptance Criteria
1. `canon init` omits Git-ignored paths from both the scan tree and file-content sections when run in a Git repository.
2. Ignored parent directories keep descendant files excluded unless the directory itself is reopened per Git semantics.
3. `--include` can still force an otherwise ignored path into the scan.
4. Existing non-Git `.gitignore` behavior continues to work as a fallback.
5. Unit tests cover the newly fixed ignore cases.
6. Manual positive and negative CLI checks are executed before sign-off.

## Verification Commands
1. `gofmt -w internal/canon/init.go internal/canon/init_test.go`
2. `go test ./...`
3. Positive CLI check:
   `go run ./cmd/canon init --root <git-fixture> --response-file <response.json> --no-interactive`
4. Negative CLI check:
   `go run ./cmd/canon init --root <git-fixture> --ai off --response-file <response.json>`
