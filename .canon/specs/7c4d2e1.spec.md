# Lint Fix Draft Spec

## Problem Statement
The Go CLI codebase has linting and style drift that should be corrected with minimal risk and no intended behavior changes.

## Scope
- In scope: lint/style fixes in `cmd/canon` and `internal/canon`.
- Out of scope: new features, refactors not required by lint, behavior changes unrelated to lint compliance.

## Assumptions
- Effective lint baseline for this repo is:
  - `gofmt -l` (formatting)
  - `go vet ./...` (vet checks)
  - `go test ./...` (safety regression gate)
- If `golangci-lint` is available in environment, run it and use `--fix` where applicable.

## Plan
1. Capture baseline with formatting/vet (and optional golangci-lint).
2. Apply auto-fixes first (`gofmt -w`, linter auto-fix if available).
3. Apply minimal manual changes for any remaining findings.
4. Add/update targeted unit tests only if manual logic-level changes are needed.
5. Verify with lint + unit tests.
6. Run one positive and one negative CLI end-to-end command relevant to touched behavior.

## Acceptance Criteria
- `gofmt -l` reports no files.
- `go vet ./...` passes.
- `go test ./...` passes.
- Positive and negative CLI command checks executed and recorded.
- Changes are limited to lint/style cleanup and any strictly required companion tests.

## Risks / Tradeoffs
- Some lints can force structural changes; minimize by preferring auto-format and narrow manual edits.
- Environment may not have optional linters installed; treat as best-effort and use Go-native checks as baseline.
