# Semantic Diff Explainer Draft Spec

## Problem Statement
Canon currently does not explain the semantic intent of code changes from repository diffs. Teams can view line-level patches with git, but still need fast, structured reasoning about likely behavioral impact, migration risk, and operational consequences.

## Desired Behavior
Add a new AI-first command, `canon semantic-diff`, that reads diff input (default `git diff`, optional `--diff-file`) and emits a deterministic semantic explanation suitable for humans and CI automation.

## Scope (v1)
In scope:
1. New CLI command `semantic-diff`.
2. Diff acquisition from:
   - default `git diff --no-ext-diff --unified=0 -- .`
   - optional `--diff-file <path>` override.
3. AI-first semantic explanation with two runtime modes:
   - `--ai auto` (default): execute provider runtime.
   - `--ai from-response`: deterministic replay via `--response-file`.
4. Structured JSON result and stable text rendering.
5. Deterministic normalization/deduplication/sorting of explanations.
6. Unit tests and CLI tests for success and failure paths.

Out of scope:
1. AST-accurate proof of runtime behavior.
2. Language-specific static analysis engines.
3. Automated code modifications or remediation patches.

## CLI Contract
Command: `canon semantic-diff`

Flags:
1. `--root <path>`: repository root (default `.`).
2. `--diff-file <path>`: read unified diff from file instead of `git diff`.
3. `--json`: emit machine-readable JSON.
4. `--ai <mode>`: `auto` or `from-response` (default `auto`).
5. `--ai-provider <name>`: `codex` or `claude` (default from config).
6. `--response-file <path>`: AI JSON replay input; required when `--ai from-response`; when provided with `--ai auto`, mode implicitly becomes `from-response`.

Validation:
1. Reject positional arguments.
2. Reject unsupported `--ai` values.
3. Reject `--response-file` missing when mode resolves to `from-response`.
4. Return error when no diff content is available.

## Output Model
`SemanticDiffResult` includes:
1. Scan context:
   - absolute root
   - diff source (`git` or file path)
   - changed files metadata
   - hunk/line stats
2. Explanations:
   - stable `id`
   - category
   - impact level (`low`, `medium`, `high`, `critical`)
   - concise summary
   - rationale
   - evidence references (file + hunk/file-level context)
3. Summary:
   - total explanations
   - highest impact
   - category counts
   - impact counts

## AI Contract
AI response schema (strict JSON object):
1. `model: string`
2. `summary: string`
3. `explanations: []`
   - `id: string`
   - `category: string`
   - `impact: low|medium|high|critical`
   - `summary: string`
   - `rationale: string`
   - `evidence: []`
     - `file: string`
     - `kind: file|hunk`
     - `old_start: int`
     - `old_lines: int`
     - `new_start: int`
     - `new_lines: int`

Normalization rules:
1. Trim whitespace and drop empty explanations.
2. Normalize impact to known enum; fallback to `medium` for unknown values.
3. Deduplicate by semantic signature (`category|impact|summary|file/hunk evidence`).
4. Sort by impact desc, category asc, summary asc, id asc.
5. Sort evidence by file then location.

## Runtime and Reliability
1. Reuse existing provider readiness checks (`codex`/`claude` presence).
2. Reuse timeout handling conventions (`CANON_AI_RENDER_TIMEOUT_SECONDS`).
3. Reuse robust JSON decoding that supports wrapped output by extracting the top-level JSON fragment.
4. Resolve relative `--response-file` and `--diff-file` paths against `--root`.

## Risks and Tradeoffs
1. AI inference may over/under-state impact; mitigated by strict schema + deterministic replay.
2. Large diffs can reduce answer quality; mitigated by concise changed-file metadata and raw diff truncation controls (future enhancement).
3. Non-unified/invalid diff files may parse partially; v1 should tolerate partial metadata extraction and still provide best-effort explanation if AI response is valid.

## Acceptance Criteria
1. `semantic-diff` produces text output with summary + explanation details.
2. `semantic-diff --json` emits valid JSON using typed result structs.
3. `--response-file` replay is deterministic and testable in CI.
4. Invalid CLI arguments are rejected with clear errors.
5. Unit tests cover diff ingestion/parsing, normalization, and error paths.
6. CLI tests cover positive replay output (text + JSON) and negative validation paths.

## Validation Plan
1. `gofmt -w` on changed Go files.
2. `go test ./...`.
3. `go vet ./...`.
4. Positive manual CLI check:
   - `go run ./cmd/canon semantic-diff --root . --diff-file <fixture.diff> --response-file <fixture-response.json>`
5. Negative manual CLI check:
   - `go run ./cmd/canon semantic-diff --root . --ai invalid`

## AI Enhancements

This spec introduces an AI-first semantic diff explainer command for Canon. The command reads repository diff input from either git diff by default or an explicit diff file, then returns structured explanations of behavioral and semantic intent. The output includes deterministic metadata about changed files, line deltas, and hunks, and produces normalized explanations with impact levels, rationale, and evidence references. The command supports auto provider execution and from-response deterministic replay so CI and offline verification can reproduce the same adjudication without rerunning providers. The implementation must enforce strict argument validation, reject unsupported AI modes, reject invalid positional usage, and fail when diff input is empty or unavailable. The runtime path must follow existing provider readiness and timeout conventions and use robust JSON fragment decoding for model output. The CLI must support text and JSON rendering with stable ordering, and tests must cover parser behavior, normalization and deduplication, deterministic ordering, response-file replay, and expected negative validation paths.
