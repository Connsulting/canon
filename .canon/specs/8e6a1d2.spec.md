# Canon Init Symlink Scan Stability

## Problem

`canon init` can fail while scanning project files when it encounters symlink entries that are not normal readable files (for example, symlinked directories or broken symlinks). The current scanner attempts to `ReadFile` every non-directory entry and aborts on read errors, which breaks initialization for repositories that legitimately contain symlinks.

## Desired Behavior

`canon init` should remain robust in repositories containing symlinks:

1. Symlink entries should not crash project scanning.
2. Symlink entries that are not guaranteed-safe regular files should be skipped from init context.
3. Initialization should still complete and produce scan context from regular files.

## Scope

In scope:

1. Scanner behavior in `scanProjectForInit`.
2. Unit tests that cover symlink handling.
3. CLI verification that demonstrates one successful run with symlinks and one expected failure case unrelated to this fix.

Out of scope:

1. Following symlink targets outside repository boundaries.
2. Expanding support to ingest symlinked content intentionally.

## Implementation Plan

1. Detect symlink entries during `WalkDir` traversal using entry mode bits.
2. Count skipped symlinks as excluded scan files.
3. Continue traversal without treating symlink read failures as fatal.
4. Add unit tests that include:
   - symlinked directory entry
   - broken symlink entry
   - confirmation that regular files are still included

## Success Criteria

1. `scanProjectForInit` does not return an error when symlinks are present.
2. Init context includes normal project files (for example `README.md`) even when symlinks exist.
3. New unit tests pass.
4. Manual CLI checks pass:
   - Positive: `canon init` succeeds in a repo with symlinks.
   - Negative: `canon init` still rejects invalid AI mode input.
