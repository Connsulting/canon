# Canon Blame

## Problem

A developer looks at the application and sees a behavior: "Why does the session expire after 15 minutes?" A product manager asks: "Where did the requirement for email verification come from?" A new team member wants to understand: "Why is the API rate-limited to 100 requests per minute?"

These are provenance questions. In code, `git blame` answers "who changed this line and when." But for specs, the question is different. It's not about a specific line in a file — it's about a *behavior* or *requirement* expressed in natural language. The answer lives somewhere in the spec corpus, possibly spread across multiple specs, possibly refined through several iterations.

Today, answering this requires manually reading specs, searching for keywords, and piecing together which spec introduced the requirement. With a large spec corpus, this is impractical.

## Proposed Solution

Add a `canon blame` command that takes a natural language description of a behavior or requirement and uses AI to trace it back to the spec(s) that introduced or mandated it.

```
canon blame "sessions expire after 15 minutes of inactivity"
```

The AI reads the full spec corpus, identifies which spec(s) contain requirements that produce the described behavior, and reports them with relevant excerpts.

## Requirements

### Core Behavior

- `canon blame "<description>"` must accept a natural language description of a behavior, requirement, or system characteristic.
- The command must load all canonical specs from `.canon/specs/`.
- The command must use AI to analyze the description against the spec corpus and identify which spec(s) are responsible for the described behavior.
- Output must include:
  - The spec ID(s) that introduce or mandate the behavior.
  - The spec title(s).
  - The specific lines or sections from each spec that establish the requirement.
  - A confidence indicator (high/medium/low) reflecting how directly the spec addresses the described behavior.
- When multiple specs contribute to a behavior, all must be listed, ordered by relevance.
- When no spec matches the described behavior, say so clearly: the behavior is either unspecified or comes from an implementation decision not captured in specs. This is itself a valuable finding.

### Output Format

- Default output must be human-readable terminal output.
- Each result must show:
  - Spec ID and title.
  - Confidence level.
  - The relevant excerpt(s) from the spec body.
  - The spec's domain and creation date for context.
- Support `--json` for machine-readable output.

### Filtering

- `--domain <name>` restricts the search to specs in a specific domain. Useful when the user already knows the general area.
- Without `--domain`, all specs are searched.

### AI Integration

- Blame must use the configured AI provider (from `.canonconfig`).
- The AI prompt must include:
  - The user's behavior description.
  - The full text of all in-scope canonical specs (or a relevant subset if the corpus is too large for the context window).
  - Instructions to identify spec(s) whose requirements would produce the described behavior.
  - Instructions to extract the specific lines that establish the requirement.
  - Instructions to assess confidence (high: spec explicitly states this; medium: spec implies this; low: behavior could be inferred but is not directly stated).
- Support `--ai-provider <name>` override.
- Support `--response-file <path>` for pre-computed AI responses.

### Context Window Management

- For large spec corpora that exceed AI context limits, blame must use a two-phase approach:
  1. **Narrow phase**: Use the index and domain metadata to select the most relevant specs (keyword overlap, domain relevance, touched-domains).
  2. **Deep phase**: Send the narrowed set to AI for detailed analysis.
- The narrowing heuristic must prefer over-inclusion (false positives are acceptable; false negatives are not).

### Unspecified Behavior Detection

- When AI determines no spec covers the described behavior, the output must clearly state this.
- This serves as a **gap detector**: behaviors that exist in the product but aren't captured in any spec represent specification gaps.
- Suggest the user consider authoring a spec to cover the gap.

## CLI Interface

```
canon blame "<description>" [options]

Options:
  --root <path>           repository root (default: .)
  --domain <name>         restrict search to this domain
  --json                  output machine-readable JSON
  --ai-provider <name>    AI provider override
  --response-file <path>  pre-computed AI response
```

## Non-Goals

- Blaming rendered state lines. This operates on behavior descriptions, not on specific lines in `state/` output. Rendered-state blame is a separate future feature.
- Tracing behavior to code. Canon tracks specs, not implementation. Code-level blame is git's job.
- Blame without AI. The natural-language matching requirement makes this inherently an AI feature. There is no deterministic fallback — if AI is unavailable, blame must fail with a clear error.
- Caching or indexing blame results. Each blame query is stateless.

- E2E CLI test: verify `--domain` filtering restricts results appropriately.
- E2E CLI test: verify `--json` output is valid JSON with expected schema.
- E2E CLI test: verify blame fails gracefully when AI provider is unavailable.
- Unit test for context-window narrowing heuristic (specs are filtered by domain/keyword relevance before AI call).

# Canon Check

## Problem

Canon's conflict detection currently only runs at `ingest` time, comparing a single candidate spec against the existing corpus. This works for linear, single-branch workflows. But canon lives inside a git repo, and teams use git branches.

When two people on separate git branches each ingest specs, git merges cleanly — every spec is its own file, so there are no file-level merge conflicts. But the *content* of those specs can semantically contradict each other. Spec A says "authentication must be required for all API endpoints." Spec B says "public endpoints must not require authentication." Git sees two different files. Canon currently has no way to notice the contradiction after the merge.

This is the spec-layer equivalent of a merge conflict, and it is invisible to git. Without `canon check`, the only way to discover these contradictions is to notice broken rendered state or stumble on them manually.

## Proposed Solution

Add a `canon check` command that scans all canonical specs for pairwise semantic conflicts. It reuses the existing normative-statement extraction and conflict detection engine (`conflicts.go`) but applies it across the full spec corpus rather than against a single candidate.

## Requirements

### Core Behavior

- `canon check` must load all specs from `.canon/specs/` and compare every pair of specs that share overlapping domains.
- Conflict detection must reuse the existing `extractNormativeStatements`, `normalizeStatementKey`, and polarity-comparison logic from `conflicts.go`.
- When conflicts are found, print a summary to stdout listing each conflict: the two spec IDs, the conflicting statement key, and the contradicting lines from each spec.
- Exit with a non-zero exit code when conflicts are detected.
- Exit with zero when no conflicts are detected.
- Print a clean confirmation message when no conflicts are found (e.g., `check passed: 0 conflicts across N specs`).

### Output Format

- Default output must be human-readable, suitable for terminal use.
- Each conflict must show:
  - The two spec IDs involved.
  - The spec titles (for human context).
  - The domain(s) where they overlap.
  - The normalized statement key.
  - The contradicting line from each spec.
- Conflicts must be grouped by spec pair for readability.
- Support a `--json` flag that outputs machine-readable JSON for CI integration.

### Filtering

- Support `--domain <name>` to restrict checking to specs in a specific domain.
- Support `--spec <id>` to check a single spec against all others (useful for targeted validation after cherry-pick or manual edits).

### CI Integration

- The non-zero exit code on conflict makes `canon check` usable as a CI gate.
- A team should be able to add `canon check` to their CI pipeline so that a git merge that introduces semantic spec contradictions fails the build.
- JSON output (`--json`) enables structured processing by CI tooling.

### Conflict Reports

- When `--write` is passed, write conflict reports to `.canon/conflict-reports/` using the existing `writeConflictReport` format.
- Without `--write`, only print to stdout (no side effects).

## Architecture

### Reuse of Existing Infrastructure

The conflict detection engine in `conflicts.go` already implements:
- `extractNormativeStatements(spec)` — extracts requirement-bearing lines with polarity.
- `normalizeStatementKey(line)` — normalizes lines into comparable keys.
- `isNormativeLine(line)` — identifies requirement-bearing sentences.
- `isNegativeLine(line)` — determines statement polarity.
- `domainsOverlap(left, right)` — checks if two specs share domains.
- `detectSpecConflicts(existing, candidate)` — detects conflicts between a candidate and existing specs.

`canon check` must generalize `detectSpecConflicts` from "one candidate vs. many" to "all-vs-all." The simplest approach: iterate over every spec as a candidate against all other specs, then deduplicate symmetric pairs (A-vs-B and B-vs-A).

### New Code

- `internal/canon/check.go` — core `Check` function.
- `CheckResult` type added to `types.go`.
- `CheckForCLI` wrapper added to `api.go`.
- CLI handler `cmdCheck` added to `cmd/canon/main.go`.

## CLI Interface

```
canon check [options]

Options:
  --root <path>     repository root (default: .)
  --domain <name>   restrict to specs in this domain
  --spec <id>       check only this spec against all others
  --json            output machine-readable JSON
  --write           write conflict reports to .canon/conflict-reports/
```

## Example Output

### No conflicts
```

```

### Conflicts found
```
conflict: b98ed16 <-> a3f7c01
  specs: "AI Render Effective State Compression" <-> "Public API Access Policy"
  domain: api
  key: api rate limiting enforced
  b98ed16: "API rate limiting must be enforced on all endpoints"
  a3f7c01: "Public API endpoints must not enforce rate limiting"

conflict: b98ed16 <-> d92ee50
  specs: "AI Render Effective State Compression" <-> "Performance Requirements"
  domain: state-render
  key: render output deterministic
  b98ed16: "Render output does not need exact byte-for-byte determinism"
  d92ee50: "Render output must be deterministic and reproducible"

check failed: 2 conflicts across 12 specs
```

### JSON output (`--json`)
```json
{
  "passed": false,
  "total_specs": 12,
  "total_conflicts": 2,
  "conflicts": [
    {
      "spec_a": "b98ed16",
      "title_a": "AI Render Effective State Compression",
      "spec_b": "a3f7c01",
      "title_b": "Public API Access Policy",
      "domain": "api",
      "statement_key": "api rate limiting enforced",
      "line_a": "API rate limiting must be enforced on all endpoints",
      "line_b": "Public API endpoints must not enforce rate limiting"
    }
  ]
}
```

## Non-Goals

- AI-assisted conflict detection. `canon check` uses the deterministic normative-statement engine only. AI-assisted checking is a future enhancement.
- Automatic conflict resolution. `canon check` reports; humans (or future `canon resolve`) fix.
- Cross-repo checking. Canon operates within a single git repo.
- Replacing ingest-time conflict detection. `canon check` is complementary — ingest catches conflicts at write time, check catches conflicts that slip in through git merge.

## Validation

- Unit tests for all-vs-all conflict detection with known contradicting specs.
- Unit tests confirming symmetric deduplication (A-vs-B is not reported twice as B-vs-A).
- Unit tests for `--domain` and `--spec` filtering.
- Unit tests for JSON output format.
- E2E CLI test: ingest non-conflicting specs, verify `canon check` exits 0.
- E2E CLI test: ingest specs with semantic contradictions, verify `canon check` exits non-zero and reports them.
- E2E CLI test: verify `--write` produces conflict report files in `.canon/conflict-reports/`.

# Canon GC

## Problem

Canon's ledger is append-only. Specs are immutable once ingested. This is correct by design — it preserves full provenance and audit history. But over time, a domain accumulates dozens of specs that overlap, partially supersede each other, or refine the same concepts through iteration.

Consider a real scenario: a team iterates on their API design over six months. They now have 23 specs touching the `api` domain. Some are original features, some are refinements, some are resolutions to conflicts between earlier specs. The rendered state handles this through AI compression, but the spec corpus itself becomes unwieldy. Reading the specs directly means piecing together intent across 23 files. Onboarding a new team member means they have to understand the full history just to know what the current requirements are.

Git has `gc` for object compaction and `squash` for commit consolidation. Canon needs the equivalent: a way to consolidate a set of specs into fewer, clean, authoritative specs that represent the current effective intent — while preserving the originals as archived history.

## Proposed Solution

Add a `canon gc` command that consolidates specs within a domain (or across specified specs) into a smaller set of clean, authoritative canonical specs. The originals are archived, not deleted — maintaining full provenance.

## Requirements

### Core Behavior

- `canon gc --domain <name>` must load all specs in the specified domain and consolidate them into one or more clean specs that capture the current effective intent.
- Consolidation must be AI-driven. The AI receives all specs in the domain and produces a minimal set of canonical specs that preserve all active requirements while eliminating redundancy, resolving superseded content, and removing stale intent.
- The original specs must be archived, not deleted. Archival means:
  - Original spec files are moved from `.canon/specs/` to `.canon/archive/specs/`.
  - Original source files are moved from `.canon/sources/` to `.canon/archive/sources/`.
  - Original ledger entries remain untouched in `.canon/ledger/` (the ledger is always append-only).
- New consolidated specs are written to `.canon/specs/` with fresh IDs and ledger entries.
- Each consolidated spec must include a `consolidates` field in its frontmatter listing the IDs of all original specs it replaces.
- The `depends_on` graph of consolidated specs must preserve any external dependencies (dependencies on specs outside the consolidation set).

### Dry Run

- Without `--write`, `canon gc` must print the proposed consolidation plan: which specs would be archived, what the consolidated specs would look like, and how many specs the domain would go from/to.
- With `--write`, execute the consolidation.

### Scope Control

- `--domain <name>` consolidates all specs in a domain. This is the primary use case.
- `--specs <id1,id2,...>` consolidates a specific set of specs (regardless of domain). Useful for targeted cleanup.
- At least one of `--domain` or `--specs` is required. `canon gc` without a scope target is an error.
- `--min-specs <n>` sets the minimum number of specs in a domain before gc considers it worth consolidating (default: 5). If the domain has fewer specs than this threshold, gc prints a message and exits cleanly. Overridable with `--force`.

### AI Integration

- Consolidation must use the configured AI provider (from `.canonconfig`).
- The AI prompt must instruct the model to:
  - Preserve all active, non-superseded requirements.
  - Resolve contradictions by honoring the most recent spec's intent (using `created` timestamps).
  - Eliminate redundant or restated requirements.
  - Produce well-structured specs with clear sections.
  - Retain domain and type metadata.
- Support `--ai-provider <name>` override, consistent with `ingest` and `render`.
- Support `--response-file <path>` for offline/pre-computed AI responses, consistent with existing AI workflows.

### Archive Structure

```
.canon/
├── archive/
│   ├── specs/       # Archived original spec files
│   └── sources/     # Archived original source files
├── specs/           # Active canonical specs (post-gc, consolidated)
├── ledger/          # Untouched — append-only history preserved
├── sources/         # Active source files
└── conflict-reports/
```

- `canon init` must create `.canon/archive/specs/` and `.canon/archive/sources/`.
- Archived files keep their original filenames.

### Ledger Entries

- Each consolidated spec gets a new ledger entry.
- The ledger entry's `parents` field must reference the most recent head(s) at the time of consolidation, maintaining DAG continuity.
- A new ledger entry type or annotation should indicate this was a gc operation (e.g., a `gc: true` field or a `type: consolidation` marker on the ledger entry).

### Safety

- `canon gc` must run `canon check` logic before consolidation. If there are unresolved semantic conflicts in the target set, gc must refuse and report them. Consolidation of conflicting specs without explicit resolution is not allowed.
- The `--force` flag overrides the min-specs threshold but does NOT override conflict safety.

## CLI Interface

```
canon gc [options]

Options:
  --root <path>           repository root (default: .)
  --domain <name>         consolidate all specs in this domain
  --specs <id1,id2,...>   consolidate specific specs by ID
  --write                 execute consolidation (default: dry run)
  --min-specs <n>         minimum specs to trigger gc (default: 5)
  --force                 override min-specs threshold
  --ai-provider <name>    AI provider override
  --response-file <path>  pre-computed AI response
```

## Non-Goals

- Automatic scheduled gc. This is always user-initiated.
- Cross-domain consolidation in a single gc pass. Each `--domain` call handles one domain. Use `--specs` for targeted cross-domain work.
- Deleting ledger history. The ledger is always append-only. Archived specs are traceable through ledger entries forever.
- Consolidating specs that have unresolved conflicts. Fix conflicts first (`canon check`), then gc.

- Unit tests for conflict-safety gate (gc refuses when conflicts exist in target set).
